## 一 · 架构与组织

### Clean Architecture

分层架构, 关键约束是 **单向依赖**: 内层不能 import 外层. Deepulse 落地为 4 包: domain (最内, 零依赖) → application → infrastructure → apps. 改外层 (换 LLM 厂商 / 换 DB) 不动内层.

### Port / Adapter (端口与适配器)

- **Port** = `application/ports/*` 里的 interface, 定义"系统需要什么" (`IBrain.stream(msg)` / `ITts.synthesize(text)`)
- **Adapter** = `infrastructure/*` 里的具体实现 (OpenAI compat / Claude CLI / VoxCPM / SoVITS)
- 价值: 同一个 Port 可以塞 N 个 Adapter, 业务代码不感知厂商差异

### 单向依赖

依赖关系是 DAG, 不能有环. apps → infrastructure → application → domain, 不能反过来. `tools/arch-test/` 跑测试强制检查这条规则.

### Monorepo

一个仓库装 N 个子项目 (apps + packages). Deepulse 有 6 个: 2 apps + 4 packages. 共享依赖 / 共享类型 / 一次 typecheck 全过.

### pnpm-workspace.yaml

告诉 pnpm "去哪些路径找子项目" (`apps/*` / `packages/*`). pnpm 自己扫这些路径里的 `package.json` 把它们识别成子包.

### catalog (pnpm)

`pnpm-workspace.yaml` 里的 `catalog:` 段, 集中声明所有第三方库版本. 子项目用 `"react": "catalog:"` 引用, 不写具体版本. 避免子项目间版本飘移.

### turbo

Monorepo 任务编排器. 跑 `pnpm dev` / `pnpm build` 时按子项目依赖关系**最大并行 + 排序 + 缓存** (没改的子项目跳过). 用 DAG 算谁能并发.

### 横切 vs 分层 (简历亮点提取用)

- **分层亮点** = "实现了 LLM 接入 / 实现了 TTS / 实现了 WebSocket / 实现了播放" (4 条, 每条单薄)
- **横切亮点** = "设计跨前后端实时音频 pipeline: SSE → sentence split → 并行 TTS → WS → 顺序播放 + ducking" (1 条立住全栈)
- 推荐写横切

## 二 · 4 个包各自管什么

### domain

**业务词汇表**, 零外部依赖. 只声明 type, 不写函数. 包含 Song / Bubble / Plan / Mood / TasteDocument / Branded ID / DomainError 等纯结构. 全栈共享同一套类型.

### application

**用例 + 端口**. 端口 (Port) 定义系统需要外部提供什么 (IBrain / ITts / INcm / IClock / 各种 Repo), 用例 (Use Case) 编排业务流程 (run-dj-turn / complete-qr-login / refresh-user-snapshot). 不依赖 infra, 只依赖 domain.

### infrastructure

**适配器集合**. 实现 application 的 Port (`brain/openai-compat` / `brain/claude` / `tts/voxcpm` / `tts/gpt-sovits` / `tts/mock` / `ncm` / `db/*` repos / `short-term-memory`). 跟外部世界对接的脏活在这.

### shared

**跨包共享但不属于业务**: env config (zod 校验启动参数) / dj-ws 协议 schema (zod) / logger / `z` re-export. 没业务概念, 纯传输层 + 配置层.

## 三 · TypeScript 与类型安全

### zod

运行时校验 + 类型推断库. 解决 TS 编译完没类型导致的"脏外部数据进系统崩"的问题. 项目里 4 处用: NCM 响应 / WS 消息 / env / HTTP body.

### z.infer · 单一真相源

先定义 zod schema, 再 `type X = z.infer<typeof xSchema>` 推导 TS 类型. 改 schema 一处, 类型 + 校验同步变. **反面教材**: 手写 `type X` + 手写 `as X` 断言, 两边不同步是经典 bug 源.

### Branded ID

```ts
type Brand<T, B extends string> = T & { readonly __brand: B }
type SongId = Brand<string, 'SongId'>
type UserId = Brand<string, 'UserId'>
```

SongId / UserId 都是 string 但 TS 编译期区分. 防你把 `userId` 传给 `play(songId: SongId)` 这种 bug.

### readonly · 不可变

domain 里所有字段都 `readonly`. 改对象要返回新对象 (`{...old, name: 'new'}`), 不能 `old.name = 'new'` 原地改. 防隐藏副作用.

### type vs interface

- `type` 用于联合 / 交叉 / 推导 (`'a'|'b'`)
- `interface` 用于可扩展的对象形状 (`extends`)
- 项目里大多用 `type`, 因为 zod 推导出来就是 type

## 四 · 网络通信

### HTTP · 客户端 / 服务端

"客户端" 跟 "服务端" 是**角色**不是身份, 同一进程可同时是两者. 例: server 对浏览器是 server, 对 NCM API 是 client.

### WebSocket

双向长连接, server 主动推消息给浏览器. DJ chat 走 WS (token / sentence / audio / action 流式推). 跟 HTTP 不是一回事, 但走同一端口.

### SSE (Server-Sent Events)

服务端单向推流, 文本格式. Brain (OpenAI 兼容) 用 SSE 流式吐 LLM token. server 接 SSE 流 → 转 WS 推前端.

### undici / fetch

- **undici**: Node 端 HTTP 客户端 (server 调 TTS / NCM 用). 比自带 fetch 控制更细 (超时 / 连接池 / 重试)
- **fetch**: 浏览器自带 + Node 22 自带. 前端调后端用

## 五 · 进程管理

### 子进程

父进程开的另一个独立进程, 通过 stdin / stdout / stderr 管道通信. Deepulse 在 `BRAIN_TYPE=claude` 时开 `claude` CLI 子进程跑 LLM.

### execa

比 Node 自带 `child_process` 友好的包装库. 提供 `cancelSignal` (AbortController 取消) / `maxBuffer` (防 OOM) / 等结束 + exitCode 检查.

### AbortController / SIGTERM / SIGKILL

- **AbortController**: JS 标准的"取消信号", 给子进程一个能中途叫停的把手
- **SIGTERM**: 通知子进程"请退出" (优雅), 子进程能截获做清理
- **SIGKILL**: 直接强杀 (不可截获), 用于 SIGTERM 不响应时兜底

## 六 · PWA (这项目其实是"半成品 PWA")

### PWA (Progressive Web App)

带 manifest + Service Worker + 能装到桌面的 web app. 仍跑在浏览器引擎里, 不是 native. localhost 也算 PWA 资格 (浏览器视 localhost 为安全上下文).

### manifest.json

告诉浏览器 "这网站是 app" 的 JSON, 写名字 / 图标 / 启动 URL / 显示模式. Deepulse 有, 见 `apps/pwa/public/manifest.json`.

### Service Worker (SW)

浏览器后台运行的 JS, 拦截网络请求做离线缓存 / push 通知 / 后台同步. **PWA 的灵魂**. Deepulse **没有**, 所以严格不算完整 PWA.

### icons (PWA)

`manifest.json` 里 `icons: [{src, sizes, type}]` 数组. Chrome 要求至少 1 个 ≥ 144px 才出现"安装"按钮. Deepulse 当前 `icons: []` 空, 装不了桌面.

### standalone

manifest 的 `display: 'standalone'`. 装到桌面后用独立窗口 (无浏览器 tab UI) 跑.

### 安全上下文 (Secure Context)

浏览器要求 HTTPS 才能用某些 API (SW / Notifications / Geolocation). **localhost 是例外**, 不需 HTTPS 也算安全上下文.

## 七 · 电台 与 音频

### FM / AM / SW (无线电)

- **FM** (Frequency Modulation) = 调频, 主流音乐电台
- **AM** (Amplitude Modulation) = 调幅, 现在多是新闻 / 脱口秀
- **SW** (Short Wave) = 短波, 远距离传播
- "FM" 在 AI 圈被 DeepFM (CTR 算法) 占了, 命名时避坑

### RX / TX

无线电术语. RX = Receiver (接收端) / TX = Transmitter (发送端). 业内昵称.

### Hz (Hertz)

频率单位. **双语境**: 音乐里 (440Hz = A 音) + 电台里 (FM 88.7MHz). 写法是 Hz (H 大写 z 小写), 不是 HZ.

### TTS (Text To Speech)

文本转语音. Deepulse 用 TTS 把 DJ 说的字幕变成声音. 有 3 个实现可切: VoxCPM (推荐) / GPT-SoVITS / mock (静音).

### VoxCPM

OpenBMB 开源 TTS, 跑在本地 :8001. Deepulse DJ 配音用它. 不是给桌宠 aemeath 的, 那个走 SoVITS.

### GPT-SoVITS

另一个开源 TTS, 跑在 :8000. **Deepulse 不用**, 桌宠 aemeath 用. 别搞混.

### Web Audio API

浏览器原生音频处理 API. 比 `<audio>` 强: 能做混音 / 滤镜 / 频谱分析. Deepulse 用它实现 ducking.

### GainNode / ducking

- **GainNode**: Web Audio API 的"音量节点", 串在播放链路中调音量
- **ducking**: DJ 说话时把音乐音量自动压低 (Deepulse 压到 25%), DJ 说完恢复. 电台行话, 来自模拟时代 DJ 手动推混音台

## 八 · DJ 业务

### DJ persona

DJ prompt 里给 LLM 的"角色设定". 必须保持"通用真人电台主播"口吻, 禁止出现"主人/动漫/游戏/IP/emoji" 等词 (连否定句也不许, 因为否定句也是 priming).

### inline action tag

DJ 在 token 流里嵌的结构化动作标签, 解决"LLM 自然语言输出 vs 系统需要结构化指令" 的矛盾:

```
"好的, 这就放<<play:周杰伦 稻香>>"
       ↓ 解析
{ kind: 'play', query: '周杰伦 稻香' }
```

三种 tag: `<<play:...>>` / `<<queue:...>>` / `<<next>>`.

### sentence segmenter

按句号 / 问号 / 感叹号切 LLM 流式 token 成完整句子, 触发逐句 TTS 合成. 不切就得等整段说完才合成, 延迟爆炸.

### brain.stream

IBrain 接口的核心方法. 给一组 messages, 返回 token 流. 不同 Brain 实现 (Claude CLI / OpenAI compat / Ollama / DeepSeek) 内部协议不同, 对外都是 token 流.

### BRAIN_TYPE / AI_URL / AI_KEY / AI_MODEL (env 配置, PRD-002 2026-06-14)

Brain 选 + 连接的 4 个 env 孔位:

- `BRAIN_TYPE` — `claude` / `deepseek` / `ollama` / `openai-compat` / `custom`, 决定走哪条**代码路径**
- `AI_URL` — LLM endpoint URL, 所有非 claude brand 共用
- `AI_KEY` — API key, 所有非 claude brand 共用 (Ollama 不验但要非空)
- `AI_MODEL` — model 名 (e.g. `deepseek-chat` / `qwen2.5:7b` / `gpt-4o-mini`)

**通用三孔哲学** (PRD-002): 所有 brand 共用同一对 env (AI_URL/AI_KEY/AI_MODEL), 不再有 brand 专属孔. BRAIN_TYPE 选 brand, 三孔填配置. 旧设计 (`DEEPSEEK_URL`/`OLLAMA_URL`/`OPENAI_BASE_URL`/`OPENAI_API_KEY` 4 孔) 已废.

### autoInferDeepseek

`shared/config/index.ts` 的兜底逻辑. 用户 set 了 `AI_KEY` 但没 set `BRAIN_TYPE` → 自动推 deepseek + 官方 AI_URL + `AI_MODEL=deepseek-chat`. 让"只 export 一个 AI_KEY" 就能跑.

## 九 · 数据库

### SQLite / better-sqlite3

- **SQLite**: 嵌入式数据库, 一个文件 = 一个库. 不需要起 server. 单用户 app 够用
- **better-sqlite3**: SQLite 的 Node 同步 API 库. 性能好, API 直接 (不用 await)

### drizzle-orm

TypeScript-first ORM. 用 TS 写 schema, drizzle 生成 SQL + 类型. 比手写 SQL 安全, 比 Prisma 轻.

### migrations

数据库 schema 演进的脚本文件. 加表 / 加字段都写一个 migration 文件, 启动时按顺序跑. drizzle-kit 自动生成.

### Repository 模式

封装数据访问, 业务代码用接口 (`ISongRepo.findById`) 不直接写 SQL. 项目里 `db/repos/*` 是具体实现.

## 十 · 命名 / 品牌相关 (昨晚讨论)

### Deepulse

项目当前名. portmanteau = Deep + **P**ulse (共享 p). 替换旧名 Claudio (抄袭嫌疑). 详见 memory [[project-name-deepulse]].

### Claudio (旧名)

原名, 词源是意大利男名 (拉丁 Claudius). 巧妙双关: Claud(e) + (au)**dio** (共享 d). 但是借鉴别人架构图上的名字, 开源前必须撇清.

### portmanteau (混合词)

两个词通过**首尾字母重叠**合成新词. 经典例子:

- breakfast + lunch = brunch
- Claude + audio = Claudio (旧名)
- Deep + Pulse = Deepulse (新名)
  不是简单拼接, 是巧妙重叠.

### SEO / 蹭流量

搜索引擎优化. 开源项目命名想火, 名字本身就得是热门搜索词 (AI / Deep / GPT). 文艺品牌词 (Nocturne / Vesper / Lyra) 没人搜.

### DeepFM (CTR 算法)

2017 IJCAI 论文, 推荐系统点击率预测的经典模型. 占用"DeepFM" 这个名字, 我们叫不了. 教训: 蹭流量前提是关键词没被强势占用.

### Deep House

电子音乐流派. 占用了"Deep + Radio" 的语义 (Deep Radio = Deep House 电台), 所以 DeepRadio 这名字撞死.

### Spotify AI DJ

Spotify 2023 推的 AI 主播功能, 用 OpenAI 写脚本 + 真人配音克隆. Deepulse "deepDJ" 候选时可蹭这流量 (后来还是选了 Deepulse).

### dub techno

Beatport 上有个艺人叫 "Deepulse" 是这个流派的. 小风险, 但他是个体不是 app.

## 十一 · 简历方法论 (备忘, 今晚不动)

### STAR

Situation 背景 / Task 任务 / Action 行动 / Result 结果. 每个简历 bullet 都隐含这 4 件.

### 技术亮点 vs 产品说明书 vs 讲故事

- **产品说明书**: 平庸, HR 看得懂工程师没感觉
- **讲故事**: 实习级可带 1 个, 资深不带
- **技术亮点** (推荐): 3-5 bullet, 每个 = 难点 + 决策 + 量化结果

### 横切 > 分层 (再强调一次, 重要)

一个 bullet 同时 cover 前后端比 4 个 bullet 分前后端立得住.
